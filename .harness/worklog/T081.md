# T081: Axiom audit: memory leaks (memory-auditor)

**Date:** 2026-07-17

## What I did

Ran the `axiom:memory-auditor` agent over the full `Sources/` tree (Tests/ has no
resource-owning types, so it contributed no candidate sites). The audit covered all six
target domains: timer leaks, NotificationCenter observer leaks, closure capture cycles,
delegate cycles, view-callback leaks, PhotoKit accumulation, and un-cancelled stored
Tasks.

## Audit result

**Zero findings at any severity (CRITICAL/HIGH/MEDIUM/LOW).** The codebase was already
clean. I independently re-read every resource-owning site the auditor flagged as
candidates, to confirm the "no issues" verdict rather than take it at face value:

| Site | Pattern | Verdict |
|---|---|---|
| `Sources/Services/AudioService.swift:18-22,40-44` | NotificationCenter observer (AVAudioSession interruption) stored as token, closure captures `[weak self]`, `deinit` calls `removeObserver` | **False positive / already correct** — matched add/remove pair, weak capture confirmed by reading the file directly |
| `Sources/Features/Browse/BrowseViewModel.swift:21-29` | NotificationCenter observer (`.entitlementsChanged`) stored as token, `[weak self]` in closure, `deinit` removes observer | **False positive / already correct** — same pattern, verified directly |
| `Sources/Features/Onboarding/WelcomeView.swift:9,99,120-121` | Stored `Task<Void, Never>?` (`restoreMessageTimer`) | **False positive** — `struct View` (value type, no retain-cycle/deinit concern), and the Task is explicitly `.cancel()`-ed at both call sites (line 99, 120) before every reassignment, so it never accumulates |
| `CardStackHero`, `PulsingAudioButton`, `AnswerRow`, `DebugMenuView`, `PlanView`, `RootView` — unstored fire-and-forget `Task {}` capturing `self` | Task capture in view body | **False positive** — all are `struct View`s; `self` capture in a value type is not a reference-counted cycle |
| No `Timer.scheduledTimer`/`Timer.publish` anywhere in `Sources/` | Timer leak | **N/A — pattern not present** |
| No `PHImageManager`/`PHCachingImageManager` usage | PhotoKit accumulation | **N/A — pattern not present** |
| No `var ... delegate:` properties anywhere in `Sources/` | Delegate cycle | **N/A — pattern not present** |
| `AppState` (root `@Observable` singleton, owns `AppServices` for app lifetime) | Long-lived singleton | **False positive / intentional** — app-root object, holds no timers/observers/tasks of its own |

No deferred findings — every reported/candidate site above is either confirmed clean or
the pattern doesn't occur in this codebase.

None — no confirmed findings required a code change.

## What remains

Nothing outstanding for this domain. Per the auditor's long-term recommendation: if a
future feature introduces `Timer`, a new `NotificationCenter` observer, `PHImageManager`,
or a delegate protocol, follow the same pattern already established here (token storage +
`deinit` teardown + `[weak self]`) to keep the codebase's memory lifecycle consistent.
