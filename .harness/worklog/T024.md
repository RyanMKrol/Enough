# T024 — StudyService (session orchestration facade)

## 2026-07-16

Implemented `Sources/Services/StudyService.swift`, a facade over `ContentStore`,
`CardSRSStore`, `ActivityStore`, and `EntitlementStore` that assembles `SessionEngine`
instances for the four study modes (learn, review, practice, learn-more) and commits
their graded results back to persistence + activity logging.

### Divergence from the sketch

The `init` takes an `entitlements: EntitlementStore` parameter in addition to
`content`, `srsStore`, `activityStore`, and `dateProvider`. This is beyond what any
architecture-doc sketch implied for this service, but is required: `makeReviewSession`
must restrict due-card lookup to owned decks (`entitlements.ownedDeckIds(catalog:)`),
and `makeLearnMoreSession` must only pull new cards from owned decks. Both behaviors
are exercised directly in the test suite (`testReviewSessionOrdersByMostOverdueAndCommits`,
`testLearnMoreSessionOnlyPullsFromOwnedDecks`).

### Implementation notes

- New-card detection: a card is "new" if `CardSRSStore.record(deckId:cardId:)` returns
  nil, or the existing record's `statusRaw == "new"`. Card order always follows
  `ContentStore.cards(forDeck:)` (the deck file's array order) — never shuffled.
- `commit(_:)` iterates `engine.gradedResults` (empty for `.practice`), computes the
  next `SRSState` via `SRSEngine.apply(grade:to:now:)`, and persists via
  `CardSRSStore.upsert(deckId:cardId:statusRaw:...)` — note `CardSRSStore` exposes a
  field-based upsert, not an `upsert(_ record:)` taking a `CardSRSRecord`, so `commit`
  fetches-or-defaults the prior `SRSState` (via `SRSBridge.state(from:)` /
  `SRSState.newCard`) and writes the new state's fields directly through that upsert
  API rather than mutating a fetched `CardSRSRecord` in place with `SRSBridge.apply`.
- One `ActivityStore.addActivity` call per commit: review mode logs `cardsReviewed`,
  learn mode logs `cardsLearned`, practice logs zero counts but still logs
  `secondsStudied` (rounded session duration).
