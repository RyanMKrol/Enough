# T022 — SRS persistence bridge + DeckProgressService

## 2026-07-16

Added `Sources/Services/SRSBridge.swift`: `SRSBridge.state(from:)` maps a `CardSRSRecord`
into an `SRSState` (unknown `statusRaw` falls back to `.new` via `CardStatus(rawValue:)`),
and `SRSBridge.apply(_:to:reviewedAt:)` writes an `SRSState` back onto a record and stamps
`lastReviewedAt`.

Added `Sources/Services/DeckProgressService.swift`: `DeckProgress` (per-deck total/learned/
newCount/learning/mastered/dueNow/strength), `progress(forDeck:)`, `totalDue()`,
`totals()` (due/learning/mastered across owned decks only), and `wordsLearned()` (learned
count across owned decks only). Owned decks come from `EntitlementStore.ownedDeckIds(catalog:)`
against `ContentStore().catalog()`; per-deck records come from `CardSRSStore.records(forDeck:)`;
`total` comes from the catalog's `DeckInfo.cardCount`. Strength follows the exact rule order
from the spec: 0 when nothing learned, 3 when nothing due, 1 when due*2 > learned, else 2.

Added `Tests/DeckProgressServiceTests.swift` (XCTest) covering: the bridge round-trip
(including the unknown-`statusRaw` fallback), the pinned scenario from the spec (30-card
`jp-greetings` deck, 10 seeded records → learned 10 / newCount 20 / learning 2 / mastered 6 /
dueNow 4 / strength 2, plus matching `totals()`/`totalDue()`/`wordsLearned()`), each strength
tier (0/3/1/2) via a dedicated scenario, and an unowned-deck-does-not-leak-into-aggregates
check.

Checks run: `swift-format lint --recursive --strict Sources Tests` (clean),
`swiftlint lint --strict` (clean — added targeted `inclusive_language`/`large_tuple` disable
comments matching the existing pattern in `SRSQueries.swift` for the spec-mandated `mastered`
field/tuple), `xcodegen generate && xcodebuild test -project Enough.xcodeproj -scheme Enough
