## What I did

Re-attempt of T106 with widened scope covering both the presented view and its two
call sites, fixing the blank-white-screen bug end-to-end.

1. `Sources/Features/Session/MCSessionView.swift`:
   - Added `SessionEnginePresentation`, an `Identifiable` wrapper (`id: ObjectIdentifier`)
     around `SessionEngine` so `SessionEngine` (owned by `SRSKit`, out of scope) doesn't
     need to be touched to support `.fullScreenCover(item:)`.
   - Replaced the fragile `.onAppear` on the nil-branch (`Color.white` placeholder) with a
     `.task` attached to the stable `NavigationStack` root, which builds the non-optional
     `viewModel` exactly once after the environment (`services`) is available. This removes
     the code path where mutating `@State` from the branch about to be swapped out could
     leave `content` stuck rendering the blank placeholder.

2. `Sources/Features/Home/HomeView.swift`:
   - Replaced `learnSessionEngine: SessionEngine?` + `isPresentingLearnSession: Bool` with a
     single `learnSession: SessionEnginePresentation?`, and switched the learn
     `fullScreenCover` to the value-driven `.fullScreenCover(item:)` form. The cover can no
     longer be presented with a nil engine. Left the review-session presentation (boolean
     `fullScreenCover`) untouched — out of scope for this fix.

3. `Sources/Features/DeckDetail/DeckDetailView.swift`:
   - Replaced the shared `sessionEngine: SessionEngine?` + `isPresentingSession: Bool` (used
     by both the learn and practice paths) with `sessionPresentation: SessionEnginePresentation?`
     and switched to `.fullScreenCover(item:)`. Both `startLearnSession` and
     `startPracticeSession` now set the same `sessionPresentation` property, preserving
     existing behavior for practice while fixing the learn path.

`MCSessionViewModel.swift` was not touched — its constructor/`loadCurrentCard` behavior did
not need to change.
