# T079 — Axiom audit: SwiftUI navigation (swiftui-nav-auditor)

**Date:** 2026-07-17

## What I did

Ran the `axiom:swiftui-nav-auditor` agent over the whole repo (`Sources/` + `Tests/`). It
mapped the navigation architecture (4 `NavigationStack`s: `HomeView` static,
`OnboardingFlowView`/`MCSessionView`/`ReviewSessionView` path-bound; a modal-local
`NavigationStack` in `DebugMenuView`'s sheet; custom hand-rolled tab shell in
`MainShellView`, no `TabView`/`NavigationSplitView`) and reported 0 CRITICAL, 2 HIGH,
2 MEDIUM, 2 LOW findings. Notably it found **none** of the specific bug patterns the task
called out as the domain focus (navigationDestination silently broken inside lazy content,
nested NavigationStacks, route enums carrying model objects, dismiss traps in
fullScreenCover flows) — all three route enums (`HomeDestination`, `SessionRoute`,
`OnboardingStep`) carry only value IDs, are registered exactly once each in the same file
that owns their path, and every `fullScreenCover` presentation self-dismisses cleanly.

## Fixes applied

- `Sources/Features/DeckDetail/DeckDetailView.swift` — `loadContent()`'s catch block now calls
  `dismiss()` instead of silently doing nothing, so a missing/stale deck id navigates back
  rather than rendering a broken blank screen.
- `Sources/Features/Onboarding/OnboardingFlowView.swift` — added a private `advance(to:)`
  helper wrapping `path.append`, used at all three call sites, removing the duplicated
  transition logic the auditor flagged.
