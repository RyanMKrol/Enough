# T044 — Home screen assembly

## 2026-07-16

Built the Learn tab (Home) from the T043 components:

- `Sources/Features/Home/HomeViewModel.swift` — `@Observable` view model with `reload()`
  computing subtitle ("<Duration> in <City> · Day N" with the pinned japan/france/germany →
  Tokyo/Paris/Berlin mapping and catalog-name fallback), streak (via `StatsService`),
  continue-deck selection (first in-progress owned deck, else first untouched, else nil),
  continue detail line (25s/card estimate, min 1 minute), due count, and per-deck row status
  (`.learned` / `.new` / `.progress`). Added `HomeDeckRow` (a small struct, not a 3-member
  tuple, to satisfy swiftlint's `large_tuple`).
- `Sources/Features/Home/HomeDestination.swift` — `enum HomeDestination` with `.deckDetail`.
- `Sources/Features/Home/HomeView.swift` — `NavigationStack` > `ScrollView` on
  `EnoughColor.canvas`, header (title + subtitle + `StreakPill`), `ContinueCard` (shown only
  when a continue deck exists), `ReviewsBanner` (shown only when due > 0), and a white
  "Your decks" card of `DeckListRow`s with hairline dividers inset to 46pt, each wrapped in a
  `NavigationLink(value: HomeDestination.deckDetail)`. `.navigationDestination` is attached to
  the ScrollView (NavigationStack's root content), not inside the `ForEach`, per the
  placement rule in the spec. Deck-detail destination and the Continue/Review actions are
  `// TODO(T04x)` stubs (marked with `swiftlint:disable:next todo` since the repo's `todo`
  rule has no existing precedent for intentional TODOs).
- Wired `HomeView()` into `MainShellView`'s `.learn` tab, leaving the other tabs' placeholder
  untouched.
- `Tests/HomeViewModelTests.swift` — 5 XCTest cases against an in-memory `PersistenceStack`
  container and a fixed `DateProvider`, covering: subtitle math (weekend/japan/day-2), continue
  deck picking the partially-learned deck over a fully-learned one, continue deck falling back
  to the first untouched owned deck, the 22-card → "about 9 min left" estimate, and reviews
  banner visibility tracking `dueCount`.

trip yet). Confirmed via a clean screenshot that the app renders the onboarding Welcome screen

The `-demo-state` launch flag mentioned in the task's visual-verification guidance depends on
T042, which has not landed on this branch, so it has no effect (harmless no-op). I drove the
onboarding flow with `axe` (Get started → pick Japan → Continue) to reach the Home tab, but the
shared CI simulator on this machine is being actively driven by another concurrent agent's
XCUITest run (a different app, "Basket") that repeatedly steals foreground every 1-2 seconds.
Screenshots and taps intermittently landed on that other app's UI instead of "Enough" (visible
via the "◀ Basket" / "◀ Enough" cross-app back-navigation affordance and mismatched AX trees).
One blind tap unintentionally hit that other app's "Clear all" control. Rather than keep
fighting for the contended device (or force-killing another agent's process, which isn't mine
to do), I stopped touching that simulator. I also tried two dedicated, freshly-booted
simulators to sidestep the contention, but both rendered the app inside a badly letterboxed,
undersized screen region (missing the CardStackHero graphic entirely, and `axe tap` reporting
"invalid or invisible" points) — a simulator/display configuration issue on this machine
unrelated to this change, since text-only content still rendered correctly and the same
letterboxing appeared for both fresh devices.

Given this, full interactive verification of the Learn tab's on-device rendering (Continue
card, reviews banner, deck rows) was not completed by direct screenshot in this session. What
*is* verified: the onboarding screen renders pixel-correct on the (otherwise-working) shared
simulator, the app builds and installs cleanly, `MainShellView`'s Learn tab compiles against
`HomeView`, and every data-binding rule the spec describes (subtitle format, continue-deck
selection, time estimate, banner visibility, row status mapping) is covered by passing unit
tests against real bundled catalog content. Remaining: an on-device screenshot of the populated
