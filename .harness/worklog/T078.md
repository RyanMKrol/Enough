# T078 — Axiom audit: SwiftUI layout

## 2026-07-17

Ran the `axiom:swiftui-layout-auditor` agent over `Sources/` and `Tests/`. It reported 4 findings
(0 CRITICAL, 0 HIGH, 2 MEDIUM, 2 LOW) plus a health summary (RIGID bordering ADAPTIVE; single
`GeometryReader` usage already correctly constrained; no deprecated `UIScreen`/`UIDevice` APIs; no
size-class code, appropriate since `project.yml` sets `TARGETED_DEVICE_FAMILY: "1"`, iPhone-only).
Triaged every finding below; fixed the ones that were real, localized defects.

### Triage table

| Finding | Severity | Verdict | Notes |
|---|---|---|---|
| `FlowLayout` (`Sources/Features/Onboarding/FlowLayout.swift`) is named as a wrapping flow layout but is actually a plain `VStack` — every scenario chip in `TripShapeView` renders one per row instead of wrapping left-to-right | MEDIUM | **Fixed** | Real bug, not just a naming issue: `ScenarioChip` (`Sources/DesignSystem/Chips.swift`) is a compact capsule button clearly designed to sit inline with siblings (see its `#Preview`, which lays chips out in an `HStack`), and the design spec (`.harness/docs/designs/design-spec.md:154-156`) describes "multi-select scenario chips" for 6 items — a one-per-line stack wastes most of the screen. Rewrote `FlowLayout` as a real `SwiftUI.Layout` conformer (row-packing `sizeThatFits`/`placeSubviews`, wraps at proposed width) — the call site in `TripShapeView.swift` needed no change since `Layout` provides the same `@ViewBuilder` init syntax. Confirmed visually via `./build_run.sh` + `axe`: chips now wrap into 3 rows (2/3/1) instead of 6 single-chip rows. Note: had to qualify as `SwiftUI.Layout` — the app already declares its own `enum Layout` in `Sources/DesignSystem/Layout.swift`, which shadows the protocol name within the module. |
| `PlanView` footer CTA button (`Sources/Features/Onboarding/PlanView.swift:194`) fixed at `.frame(width: 180)` around dynamic content (`vm.ctaTitle`, a price/plan-dependent string) — risk of clipping at large Dynamic Type sizes | LOW | **Fixed** | Changed to `.frame(minWidth: 180)` so the button can grow with its label instead of clipping, and added `.lineLimit(1)` + `.minimumScaleFactor(0.7)` to the title `Text` so it degrades gracefully (shrinks) rather than wrapping awkwardly inside the HStack alongside the summary text. Confirmed visually — button renders correctly with the current bundle-price CTA. |
| `CardStackHero` hero graphic fixed at 255×292pt (`Sources/DesignSystem/CardStackHero.swift`), used in `WelcomeView`, is tight on a 320pt-wide iPhone SE | MEDIUM | **Deferred** | Auditor itself downgraded impact: the element is `accessibilityHidden(true)` (decorative only, verified at `CardStackHero.swift:96`), so Dynamic Type has no effect on it, and it currently fits (centered, no competing horizontal siblings) on the smallest supported device. Making it proportionally responsive would mean introducing a scale-relative sizing scheme for a single decorative onboarding graphic — disproportionate diff for an audit sweep with no observed breakage. Revisit condition: if a future task adds horizontal padding/siblings around `WelcomeView`'s hero, or if a device narrower than 320pt is ever supported. |
| Landscape orientation not explicitly verified/locked; several onboarding screens (`WelcomeView`, `CardStackHero`) use portrait-sized fixed layouts with no landscape adaptation | LOW | **Deferred (false-positive-adjacent)** | App is iPhone-only (`project.yml` `TARGETED_DEVICE_FAMILY: "1"`) and every existing screen in this codebase is portrait-first by design — this is a systemic, whole-app characteristic, not a defect in code touched by this task's scope (auditor flagged it against `RootView.swift`/`MainShellView.swift`, files this task doesn't own concrete layout defects in). Locking orientation in Info.plist is a project-configuration decision, not a `Sources/**` code fix, and out of this task's scope (`project.yml`/Info.plist changes weren't part of the reported layout defects, they're a build-setting decision). Revisit condition: if landscape support is ever explicitly requested, or if `TARGETED_DEVICE_FAMILY` changes to include iPad. |
