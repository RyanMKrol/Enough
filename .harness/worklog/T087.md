# T087 — Axiom audit: Swift performance (swift-performance-analyzer)

**Date:** 2026-07-17

## What I did

Ran the `axiom:swift-performance-analyzer` agent over `Sources/` and `Tests/`, focused on the
domains called out in the spec (ARC/copies in SRS queue/session paths, collection inefficiencies
over the catalog, unspecialized generics). Triaged every finding against this app's actual scale
(a small trip-vocab app — a handful of countries/decks, no `any`/`weak` usage anywhere in the
codebase) and fixed the ones with plausible user-visible impact (session-start/commit latency,
Home/Progress screen load scaling with number of owned decks).

## Fixes applied

- `Sources/Persistence/CardSRSStore.swift`: added `records(forDecks:)` (batched multi-deck fetch,
  same in-memory `Set.contains` filter pattern as the existing `dueRecords`) and `upsertBatch(_:)`
  (one batched fetch + one `context.save()` for N updates, via a new `CardSRSStore.Update` struct).
- `Sources/Services/StudyService.swift`: `makeLearnSession`, `makePracticeSession`,
  `makeLearnMoreSession` now build a `[cardId: CardSRSRecord]` dictionary once per deck instead of
  fetching per card; `commit(_:)` batches all graded-card updates through `upsertBatch(_:)` instead
  of fetching + saving per card.
- `Sources/Services/DeckProgressService.swift`: `totals()` and `wordsLearned()` now call
  `records(forDecks:)` once across all owned decks instead of looping `records(forDeck:)` per deck.
- `Tests/CardSRSStoreTests.swift`: added `testRecordsForDecksReturnsRowsAcrossMultipleDecks`,
  `testUpsertBatchInsertsAndUpdatesInOneSave`, `testUpsertBatchWithNoUpdatesIsNoOp` covering the
  new batched APIs directly (insert + update in one batch, multi-deck fetch, empty-batch no-op).

No pinned-behavior test was changed — all existing `StudyServiceTests`/`DeckProgressServiceTests`/
`CardSRSStoreTests` assertions pass unmodified against the refactored implementation, confirming
the fixes are behavior-preserving.
