# T084 — Axiom audit: UX flows (ux-flow-auditor)

## 2026-07-17

Ran the `axiom:ux-flow-auditor` agent over the full codebase (Sources/ + Tests/), scoped to
UX flow defects: dead ends, dismiss traps, buried CTAs, missing loading/error/empty states,
broken data paths, accessibility dead ends. Triaged every finding against the repo's
architecture and design spec, then fixed every confirmed one.

### Triage table

| # | Finding | Verdict | Notes |
|---|---------|---------|-------|
| 1 | `OnboardingFlowView.finish()` swallows `OnboardingCompleter.complete` errors — purchase can succeed while trip-save fails silently, stranding the user in onboarding | **Fixed** | `finish()` now returns `Bool`; `PlanView.purchase()` surfaces a retry-worthy `errorNote` ("...tap Continue to retry") on failure instead of swallowing it |
| 2 | `CountryPickerView.loadCatalog()` swallows catalog load failure on the first onboarding screen — grid renders empty, Continue stays disabled with no explanation | **Fixed** | Added `loadError` state + inline "We couldn't load the destinations" error view with a Try again button |
| 3 | `FlashcardView` reveal is reachable only via `.onTapGesture`, no accessibility label/trait/action — blocks VoiceOver users from the core review flow | **Fixed** | Added `.accessibilityAddTraits(.isButton)`, label, hint, and `.accessibilityAction` when not yet revealed; scoped to pre-reveal state only so the front/back audio buttons stay individually reachable once revealed (avoided `.combine` swallowing them permanently) |
| 4 | `startLearnSession`/`startReviewSession`/`startPracticeSession` silently no-op on failure across `HomeView`, `DeckDetailView`, `ReviewsTabView` (5 call sites) — primary CTA can do nothing with zero feedback | **Fixed** | Each now does `try`/`catch` and surfaces a `sessionStartError` via the new shared `.errorAlert` view modifier (`Sources/DesignSystem/ErrorAlertModifier.swift`), matching `BrowseView`'s existing alert pattern |
| 5 | `DeckDetailView.loadContent()` calls `dismiss()` on any error, including transient ones — user taps a deck and is silently kicked back to Home | **Fixed** | Replaced with a `loadFailed` state + inline "We couldn't load this deck" banner with a Try again button; no more auto-dismiss |
| 6 | `DeckDetailView.resetProgress()` swallows errors after a destructive confirmed action — UI gives no signal either way | **Fixed** | Surfaces `resetErrorMessage` via `.errorAlert` on catch |
| 7 | `DeckDetailView` has no empty-state row when `cards` is empty | **Fixed** | Added a lightweight "No cards in this deck yet" row, shown only when not also in the load-failed state |
| 8 | No `NavigationSplitView`/size-class adaptivity for iPad | **False positive** | `project.yml` sets `TARGETED_DEVICE_FAMILY: "1"` (iPhone-only); iPad adaptivity is out of scope by design |

### Fix details

- `Sources/Features/Onboarding/OnboardingFlowView.swift`, `PlanView.swift` — `finish()` returns
  success/failure; `PlanView` reuses its existing `errorNote` footer pattern.
- `Sources/Features/Onboarding/CountryPickerView.swift` — inline retryable error state.
- `Sources/Features/Session/FlashcardView.swift` — accessibility label/hint/trait/action on the
  card, conditioned on `isRevealed` to avoid combining the audio-play buttons into one element.
- `Sources/Features/Home/HomeView.swift`, `Sources/Features/Reviews/ReviewsTabView.swift`,
  `Sources/Features/DeckDetail/DeckDetailView.swift` — session-start error alerts via the new
  `Sources/DesignSystem/ErrorAlertModifier.swift` (`View.errorAlert(_:message:)`), reused across
  all three screens and `BrowseView`-style.
- `DeckDetailView` also split into `DeckDetailHeroSection` / `DeckDetailActionBar` private
  subviews to keep `swiftlint`'s `type_body_length` under the 250-line limit after adding the
  new error/empty states — pure extraction, no behavior change.
