### Implementation
Built AudioService with per-card clip playback and auto-play-once semantics:

1. **AudioService.swift**: Core playback service with:
   - `play(fileName:)` - resolves audio URL from ContentStore, invokes test seam (playHandler) or creates/retains AVAudioPlayer
   - `autoPlayOnce(cardId:fileName:)` - tracks played cards via Set, invokes play() only on first appearance per session
   - `resetAutoPlay()` - clears played set AND configures + activates shared AVAudioSession with .playback category and .duckOthers (allows pronunciation through silent switch, respects user's other audio)
   - `sessionEnded()` - stops current player and deactivates session to restore ducked audio volume
   - Interruption handling via NotificationCenter observer (block-based) on .main queue; stops clip on .began, no auto-resume

2. **AudioServiceTests.swift**: 9 comprehensive tests covering:
   - `play(fileName:)` with valid fixture → invokes playHandler exactly once with correct URL
   - `play(fileName: "does-not-exist.mp3")` → no handler invocation, no crash
   - `autoPlayOnce()` twice same cardId → plays once; different cardId → plays again (keyed by card, not file)
   - `resetAutoPlay()` → clears played set, allows replaying same card
   - `sessionEnded()` with playHandler → doesn't crash, doesn't clear played set
   - Conformance check: no AVAudioPlayerDelegate on service
   - Interruption observer removed in deinit (basic validation)

### Design notes

- Used `@preconcurrency import AVFoundation` + `@unchecked Sendable` to handle Swift 6 strict concurrency while relying on NSObjectProtocol's known-safe NotificationCenter integration
- Interruption observer registered on .main queue (safe for main-actor-isolated currentPlayer access)
- Silent ignore on missing audio files (placeholder content may lack clips) and audio player errors
- Session category deliberately `.playback` + `.duckOthers` per spec (pronunciation must audible through silent switch; other audio fades rather than stops)
