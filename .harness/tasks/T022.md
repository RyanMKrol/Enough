# T022 — SRS persistence bridge + DeckProgressService (counts, strength, dues)

## Overview

Connect the pure SRS types to the app's stores: a two-way bridge between the SwiftData
`CardSRSRecord` (T017) and `SRSState` (T019), plus `DeckProgressService` — the single source for
every per-deck/aggregate number the UI shows (learned counts, due counts, Mastered totals, and
the Reviews tab's 0–3 strength bars; design-spec.md §4, §7.1).

## Do

1. Create `Sources/Services/SRSBridge.swift`:

   ```swift
   enum SRSBridge {
     static func state(from record: CardSRSRecord) -> SRSState
     static func apply(_ state: SRSState, to record: CardSRSRecord, reviewedAt: Date)
   }
   ```

   `state(from:)` maps `statusRaw` via `CardStatus(rawValue:)` (fall back to `.new` on an
   unknown string) and copies easeFactor/intervalDays/repetitions/lapses/dueAt verbatim.
   `apply` writes every field back and sets `record.lastReviewedAt = reviewedAt`. A card with NO
   record is, by definition, `SRSState.newCard`.

2. Create `Sources/Services/DeckProgressService.swift`:

   ```swift
   final class DeckProgressService {
     struct DeckProgress: Equatable {
       let deckId: String
       let total: Int      // cardCount from the catalog DeckInfo
       let learned: Int    // records with status != .new
       let newCount: Int   // total - learned
       let learning: Int   // status == .learning
       let mastered: Int   // SRSEngine.isMastered
       let dueNow: Int     // SRSEngine.isDue at dateProvider.now
       let strength: Int   // 0–3, rule below
     }
     init(content: ContentStore, srsStore: CardSRSStore,
          entitlements: EntitlementStore, dateProvider: DateProvider)
     func progress(forDeck id: String) throws -> DeckProgress
     func totalDue() throws -> Int
     func totals() throws -> (due: Int, learning: Int, mastered: Int)  // owned decks only
     func wordsLearned() throws -> Int                                 // owned decks only
   }
   ```

3. Strength rule, evaluated in this exact order: `learned == 0` → **0**; `dueNow == 0` → **3**;
   `dueNow * 2 > learned` → **1**; otherwise → **2**.

4. Owned decks come from `entitlements.ownedDeckIds(catalog:)` (T016) against
   `content.catalog()`. Per-deck records come from `srsStore.records(forDeck:)` (T017).
   `total` comes from the catalog's `DeckInfo.cardCount`, NOT from counting records.

5. Create `Tests/DeckProgressServiceTests.swift` (XCTest). Test setup: in-memory container via
   `PersistenceStack.container(inMemory: true)` (T014); the REAL bundled catalog via
   `ContentStore()` (the unit-test bundle is hosted by the app, so `Bundle.main` contains the
   committed `Content/` fixture — deck `jp-greetings` has `cardCount == 30`); a private
   `struct FixedDateProvider: DateProvider { let now: Date }`. Seed `CardSRSRecord` rows with
   card ids read from `ContentStore().cards(forDeck: "jp-greetings")`.

## Done when

- Bridge round-trip test: build an `SRSState` (`.review`, ease 2.1, interval 8, reps 3, lapses 1,
  dueAt fixed), `apply` it to a fresh record, `state(from:)` returns an equal state, and
  `lastReviewedAt` equals the passed date. Unknown `statusRaw` `"garbage"` maps to `.new`.
- With entitlements granting `jp-greetings` and 10 seeded records (6 `.review` interval 30 due
  in the future, 2 `.learning` due 1 h AGO, 2 `.review` interval 8 due 1 h ago):
  `progress(forDeck: "jp-greetings")` == total 30, learned 10, newCount 20, learning 2,
  mastered 6, dueNow 4, strength 2; `totals()` == (due: 4, learning: 2, mastered: 6);
  `totalDue()` == 4; `wordsLearned()` == 10.
- Strength cases each pinned by a dedicated scenario: no records → 0; records but zero due → 3;
  due > half of learned (e.g. 3 of 4 due) → 1; due > 0 but ≤ half (e.g. 2 of 10) → 2.
- An UNOWNED deck's records do not leak into `totals()`/`totalDue()`/`wordsLearned()` (seed a
  second deck's records without granting it and assert the aggregates are unchanged).
