# T024 — StudyService: build & commit sessions (learn, review, practice, learn-5-more)

## Overview

The orchestration layer between content, persistence and the pure SRS machinery: it constructs
`SessionEngine`s for the four session kinds the UI launches (learn a deck, review everything due,
free practice, "Learn 5 more"), and commits a finished engine's grades back into `CardSRSRecord`s
plus a `DailyActivityRecord` row.

## Do

1. Create `Sources/Services/StudyService.swift`:

   ```swift
   final class StudyService {
     init(content: ContentStore, srsStore: CardSRSStore, activityStore: ActivityStore,
          entitlements: EntitlementStore, dateProvider: DateProvider)
     func makeLearnSession(deckId: String, size: Int = 12) throws -> SessionEngine
     func makeReviewSession() throws -> SessionEngine
     func makePracticeSession(deckId: String) throws -> SessionEngine
     func makeLearnMoreSession(size: Int = 5) throws -> SessionEngine
     func commit(_ engine: SessionEngine) throws
   }
   ```

   Note: this init adds `entitlements:` beyond the architecture.md sketch — review/learn-more
   need the owned-deck set. Record the divergence in the worklog; later tasks bind to THIS
   signature.

2. New-card detection: a card is NEW when `srsStore.record(deckId:cardId:)` returns nil OR the
   record's `statusRaw == "new"`. Card order is always the deck file's array order
   (`content.cards(forDeck:)`), never shuffled.
   - `makeLearnSession`: the deck's new cards → `SRSQueueBuilder.learnBatch(newCardIds:, size:)`
     → `SessionCard`s (state from `SRSBridge.state(from:)` or `SRSState.newCard`), mode `.learn`.
   - `makeReviewSession`: `srsStore.dueRecords(now: dateProvider.now, ownedDeckIds:)` (owned ids
     via `entitlements.ownedDeckIds(catalog: content.catalog())`) → `QueueEntry`s →
     `SRSQueueBuilder.reviewQueue(from:now:)` → mode `.review`.
   - `makePracticeSession`: ALL of the deck's cards, deck order, mode `.practice`.
   - `makeLearnMoreSession`: walk the catalog's owned decks in catalog order, collecting new
     cards until `size` (5) are found, mode `.learn`.
   - Every engine gets `now: { [dateProvider] in dateProvider.now }`.
3. `commit(_:)`:
   - For each `(card, grade)` in `engine.gradedResults` (empty for practice): fetch-or-create the
     record, `let newState = SRSEngine.apply(grade, to: SRSBridge.state(from: record), now:
     dateProvider.now)`, then `SRSBridge.apply(newState, to: record, reviewedAt:
     dateProvider.now)`, then `srsStore.upsert(record)`.
   - One activity call: `activityStore.addActivity(on: dateProvider.now, cardsReviewed: r,
     cardsLearned: l, secondsStudied: s)` where review mode → `r = gradedResults.count, l = 0`;
     learn mode → `l = gradedResults.count, r = 0`; practice mode → both 0; and
     `s = Int(engine.summary().duration.rounded())`. Practice still logs its seconds.
4. Create `Tests/StudyServiceTests.swift` (XCTest): in-memory container
   (`PersistenceStack.container(inMemory: true)`), real bundled catalog (`ContentStore()` —
   `Bundle.main` in the hosted test bundle contains the committed `Content/` fixture; deck
   `jp-greetings` has 30 cards), `struct FixedDateProvider: DateProvider { let now: Date }`,
   entitlements granted for `jp-greetings` only.
