## Overview
A human validation pass over the project's prose documentation to confirm each doc still reflects the current code, backlog, and shipped state — catching drift that an autonomous loop run can silently introduce. Produces no code change; it is a recorded review, not an edit.

## Do
Review each of these docs against the current code and backlog, judging accuracy/staleness: README.md (especially the '## What's implemented', '## Architecture', and '## Content' sections — README is the declared source of truth for current state); root CLAUDE.md (conventions/golden rules still describe how the repo actually works); .harness/custom/docs/LIMITATIONS.md (each row still holds, and check whether every 'Revisit:' trigger has now fired — e.g. StoreKit stub → T062, sample content fixture → real scenario packs, WCAG-AA text tokens → accessibility audit, Enough-Sim hardcoded UDID); and the two project design docs .harness/docs/designs/architecture.md and .harness/docs/designs/design-spec.md (signatures/design-truth still match the code). 'Review' means read the doc, spot-check its claims against the actual code/TASKS.json, and note any mismatch.

## Done when
The human has reviewed every doc named above and recorded, in the task worklog, a per-doc verdict (valid / stale) with a specific pointer for each staleness found (which line/section is wrong and what reality is) — including, for LIMITATIONS.md, whether each 'Revisit:' trigger has now fired. No documentation edits are required by this task itself; fixes (if any) are handled separately per the coordinator's decision on follow-up tasks.
