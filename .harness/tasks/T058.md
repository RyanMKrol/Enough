# T058 — New-trip flow: start/switch trips, accent reskin, owned packs persist

## Overview

Let the user finish one trip and start another: a "New trip" entry on the Progress tab that
deactivates the current trip, wipes learning progress (NEVER entitlements), and re-runs
onboarding so the app reskins to the new country's accent. Design reference:
.harness/docs/designs/design-spec.md §0 (accent theming) + §2; architecture:
.harness/docs/designs/architecture.md.

## Do

1. `Sources/Features/Progress/ProgressTabView.swift`: add a **"New trip"** text button
   top-right of the title row (13 Semibold, `EnoughColor.linkBlue`). Tapping sets a local
   `@State` that presents a `.confirmationDialog` titled **"Start a new trip?"** with message
   **"Your packs stay yours. Progress and streaks reset."**, actions **"Start new trip"**
   (role `.destructive`) and **Cancel** (role `.cancel`). The destructive action calls
   `appState.startNewTrip()` (AppState reachable per the app's established pattern —
   environment or injected; follow how `MainShellView` already gets it).
2. `Sources/App/AppState.swift`: implement `func startNewTrip()`:
   - `services.tripStore.deactivateActiveTrip()` (see step 3),
   - `services.cardSRSStore.reset()` and `services.activityStore.reset()` (wipe SRS rows +
     daily activity),
   - do NOT touch `EntitlementStore`, purchases, or the "notifications.asked" flag,
   - cancel any pending review notification via `services.notifications.cancelAll()` if the
     `notifications` service exists on `AppServices` at build time (skip this line otherwise),
   - set `phase = .onboarding`.
3. `Sources/Persistence/TripStore.swift`: add `func deactivateActiveTrip()` — sets
   `isActive = false` on the current active `TripProfileRecord` (no-op when none) and saves.
   Keep the record (trip history), don't delete it. If an equivalent already exists under a
   different name, reuse it instead of adding a duplicate.
4. Re-entry behavior in `Sources/Features/Onboarding/OnboardingFlowView.swift`: add
   `init(startAtCountryPicker: Bool = false)`. When `true`, the flow begins directly on
   `CountryPickerView` (skip `WelcomeView` — the marketing cold-open is for first launch only);
   back from the country picker is hidden/disabled in this mode (there is nowhere to go back
   to). `RootView` passes `startAtCountryPicker: true` when onboarding was re-entered via
   `startNewTrip()` (e.g. an `AppState` flag `isReonboarding`, set by `startNewTrip()` and
   cleared on completion) and `false` on first launch. Completing the flow works exactly as on
   first run: it creates the new ACTIVE `TripProfileRecord`, sets `phase = .main`, and
   `AppState.activeAccent` re-derives from the new trip so every accent-driven surface reskins.
5. Add/extend an XCTest (e.g. `Tests/AppStateTests.swift`) with an in-memory container:
   seed an active Japan trip + entitlements + SRS rows + activity rows; call `startNewTrip()`;
   assert the trip is inactive, SRS and activity stores are empty, entitlements are untouched,
   and phase is `.onboarding`.

## Done when

- The test above passes: after `startNewTrip()` — no active trip, zero SRS/activity rows,
  entitlements intact, phase `.onboarding`.
- Cancel in the dialog changes nothing.
- Verify on the simulator: launch with `-demo-state` (Japan trip, red accent, 3 owned packs) →
  Progress tab → "New trip" → confirm → onboarding opens at the country picker (no welcome
  screen); pick **France**, finish the flow → the shell returns with the BLUE France accent on
  buttons/tab bar, Home shows the new trip with everything "New"/zero progress, and Browse
  still shows the previously bought Japan packs as "Owned". Capture via `./build_run.sh`.
