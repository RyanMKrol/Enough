## Overview

Replace the stubbed purchase path with real StoreKit 2, behind the existing `PurchaseProviding`
protocol, so packs and bundles are actual non-consumable in-app purchases. T061 (needs-human)
has already created the App Store Connect products and recorded the product-id convention.

## Do

1. Read the product-id ↔ catalog-id convention T061 recorded (in T061's worklog entry under
   the harness worklog directory, and/or a §0 note in design-spec.md — read-only references).
   Default convention: `com.ryankrol.enough.<catalog-id>` (e.g. `com.ryankrol.enough.jp-weekend`).
2. Create `Enough.storekit` at the repo root — a StoreKit configuration file listing every
   product id for the catalog's decks and bundles with their GBP prices (decks £1.00,
   jp-weekend £2.49, jp-week £3.49; mirror for fr-/de- ids). Reference it from `project.yml`
   so the scheme uses it for local/simulator runs and tests
   (`schemes.Enough.run.storeKitConfiguration` in XcodeGen — the path is PROJECT-RELATIVE and
   easy to get wrong; the products-load test below exists to catch exactly that).
3. Create `Sources/Services/StoreKitPurchaseService.swift`: `final class
   StoreKitPurchaseService: PurchaseProviding`.
   - Maps catalog id ↔ product id via the convention (helper funcs, unit-tested).
   - **Init order matters:** FIRST start the `Transaction.updates` listener, THEN reconcile
     from `Transaction.currentEntitlements` (silently iterate it, skipping any transaction
     with `revocationDate != nil`, granting via `EntitlementStore`) — this recovers
     reinstall/new-device purchases with no user action and no sign-in prompt.
   - **Listener requirements** (all four are load-bearing): (a) started in `init` before
     anything else; (b) the `Task` is STORED in a property — an unretained task is silently
     dropped and external purchases never arrive; (c) cancelled in `deinit`; (d) it is the
     app's one sanctioned `Task.detached` — hop to the main actor for EntitlementStore writes
     and for posting `.entitlementsChanged`.
   - **`transaction.finish()` discipline — EVERY delivered transaction ends finished**, or the
     queue re-delivers it on every launch forever:
     - verified → grant FIRST, then `await transaction.finish()` (never finish-before-grant: a
       crash between finish and grant loses a paid unlock);
     - unverified → do NOT grant, but still finish;
     - revoked/refunded (`revocationDate != nil`) → `EntitlementStore.revoke(productId:)`,
       then finish.
   - `purchase(productId:kind:) async throws -> PurchaseOutcome`:
     `Product.products(for:)` → `product.purchase()` → map the result:
     `.success(let verification)` → only `.verified` grants (use `try result.payloadValue` or
     pattern-match; unverified grants nothing but is finished) → grant → finish → post
     `.entitlementsChanged` → return `.success`; `.userCancelled` → return `.cancelled` (a
     normal return, never a throw); `.pending` → return `.pending` and grant NOTHING (Ask to
     Buy — the updates listener delivers the entitlement later if approved).
   - `restorePurchases()`: `try await AppStore.sync()` — this is the ONLY call site of
     `sync()` in the app (it can force an App Store sign-in prompt; never call it at launch) —
     then re-derive grants from `Transaction.currentEntitlements`.
   - `displayPrice(productId:)`: return the loaded `product.displayPrice` (cache products
     after first load); fall back to the catalog/`PricingCalculator` price while products
     haven't loaded — so every storefront sees Apple's localized price.
   - `ownedProductIds` stays backed by `EntitlementStore` (single source the UI already reads).
4. If `EntitlementStore` lacks `revoke(productId:)`, this task adds it (a targeted deletion of
   the matching `EntitlementRecord`) as an extension in an in-scope file (e.g.
   `Sources/Services/EntitlementStore+Revoke.swift`) — the persistence directory itself is NOT
   in this task's scope; if an extension proves impossible without editing out-of-scope files,
   record `failed:blocked` in the worklog naming the one method needed.
5. Wire it in WITHOUT touching the App layer (AppServices' own directory is out of scope):
   expose a factory `PurchaseServiceFactory.live(entitlements:catalog:) -> PurchaseProviding`
   in `Sources/Services/PurchaseServiceFactory.swift` returning the StoreKit implementation
   (keep the stub for previews and tests). If AppServices.live() does not already route
   through such a factory, record `failed:blocked` in the worklog asking for the one-line
   App-layer hookup rather than editing out-of-scope files.
6. Tests (`Tests/StoreKitPurchaseServiceTests.swift`): product-id mapping round-trip for every
   catalog id; and — only if the CI environment supports `StoreKitTest` (import compiles) —
   `SKTestSession` tests with `SKTestSession(configurationFileNamed: "Enough")`,
   `session.disableDialogs = true` (mandatory for headless CI) and
   `session.clearTransactions()` in `setUp`:
   - REQUIRED: `Product.products(for: <all catalog product ids>)` returns one product per id —
     this catches an `Enough.storekit` file that isn't actually wired into the scheme, which
     otherwise leaves every other test vacuously green;
   - a purchase of `com.ryankrol.enough.jp-greetings` grants the deck via EntitlementStore and
     the transaction is finished.
   If `StoreKitTest` is unavailable headless, keep the mapping tests and document the manual
   sandbox check in the worklog instead — do not ship a flaky test.
