## Overview

Replace the stubbed purchase path with real StoreKit 2, behind the existing `PurchaseProviding`
protocol, so packs and bundles are actual non-consumable in-app purchases. T061 (needs-human)
has already created the App Store Connect products and recorded the product-id convention.

## Do

1. Read the product-id ↔ catalog-id convention T061 recorded (in T061's worklog entry under
   the harness worklog directory, and/or a §0 note in design-spec.md — read-only references).
   Default convention: `com.ryankrol.enough.<catalog-id>` (e.g. `com.ryankrol.enough.jp-weekend`).
2. Create `Enough.storekit` at the repo root — a StoreKit configuration file listing every
   product id for the catalog's decks and bundles with their GBP prices (decks £0.99 tier,
   jp-weekend £2.49, jp-week £3.49; mirror for fr-/de- ids). Reference it from `project.yml`
   so the scheme uses it for local/simulator runs and tests
   (`schemes.Enough.run.storeKitConfiguration` in XcodeGen).
3. Create `Sources/Services/StoreKitPurchaseService.swift`: `final class
   StoreKitPurchaseService: PurchaseProviding`.
   - Maps catalog id ↔ product id via the convention (helper funcs, unit-tested).
   - `purchase(productId:kind:)`: `Product.products(for:)` → `product.purchase()` → verify the
     `VerificationResult` (`.verified` only), `transaction.finish()`, then grant via
     `EntitlementStore` (same rows the stub writes — the rest of the app is unchanged).
   - `restorePurchases()`: `try await AppStore.sync()` then re-derive grants from
     `Transaction.currentEntitlements`.
   - A `Transaction.updates` listener task started at init grants/revokes on external changes.
   - `ownedProductIds` stays backed by `EntitlementStore` (single source the UI already reads).
4. Wire it in WITHOUT touching the App layer (AppServices' own directory is out of scope):
   expose a factory `PurchaseServiceFactory.live(entitlements:) -> PurchaseProviding` in
   `Sources/Services/PurchaseServiceFactory.swift` returning the StoreKit implementation (keep
   the stub for previews and tests). If AppServices.live() does not already route through such
   a factory, record `failed:blocked` in the worklog asking for the one-line App-layer hookup
   rather than editing out-of-scope files.
5. Tests (`Tests/StoreKitPurchaseServiceTests.swift`): product-id mapping round-trip for every
   catalog id; and — only if the CI environment supports `StoreKitTest` (import compiles) — an
   `SKTestSession` purchase of `com.ryankrol.enough.jp-greetings` grants the deck via
   EntitlementStore. If `StoreKitTest` is unavailable headless, keep the mapping tests and
   document the manual sandbox check in the worklog instead — do not ship a flaky test.

## Done when

- `Enough.storekit` exists, is referenced by the scheme, and lists every catalog product id.
- Buying a deck in the simulator (StoreKit config sandbox) unlocks it in Browse/Home, and
  "Restore purchases" re-grants after a reset via the debug menu's "Reset all data".
- Product-id mapping is unit-tested for all catalog ids; no UI file changed (protocol unchanged).
