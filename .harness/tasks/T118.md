## Overview
Idea #2: owning multiple packs (e.g. France AND Japan) offers no way to switch which one you're learning. This is a MODEL gap, not just a missing button, and it hinges on human product+data decisions that must be settled before the switcher can be specified. This is a needs-human unit: a human records the decision; the dependent follow-up implements it.

## Do
Record a written design decision (in the worklog and, if it introduces a lasting trade-off, a row in .harness/custom/docs/LIMITATIONS.md) answering these questions: (1) Does switching RESUME a previously-created trip (requires TripProfileRecord to hold multiple rows with one active + a TripStore setActive/allTrips API + preserving each trip's Day-N and SRS state), or does it only START a new trip for another owned pack (simpler, loses the old trip's day count)? (2) If multi-trip: how are per-pack SRS (CardSRSRecord) and Day-N/daily-activity (DailyActivityRecord) namespaced when trips coexist — today both are keyed by deckId ONLY, so two trips over the same country would collide; does this need a schema change (trip/country key) and a migration? (3) Where does the switcher UI live — tapping the Learn header (HomeView subtitle), a menu, or repurposing the Browse tab (currently header + Restore-purchases + pack list)? (4) The switcher must only offer packs the user actually owns (EntitlementStore). Do NOT write switcher implementation code in this unit — only settle and record the decision so the dependent follow-up can be specified from it.

## Done when
A human has recorded a clear decision covering: resume-vs-start-new, whether/how TripProfileRecord becomes multi-row and gains setActive/allTrips, how CardSRSRecord/DailyActivityRecord are namespaced across coexisting trips (incl. any schema/migration needed), and where the switcher UI lives — enough for the dependent buildable follow-up to be authored without further human input.
