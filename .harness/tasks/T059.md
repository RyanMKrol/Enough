# T059 — Motion & haptics conformance sweep against the design spec

## Overview

A cross-cutting audit-and-fix pass: verify every animation and haptic in the app matches the
design spec's motion vocabulary exactly, centralise all durations into the `Motion` constants,
and remove any stray feedback. This is conformance work ONLY — constants, easings, and haptic
calls; no new features, no renames, no layout changes. Design reference:
.harness/docs/designs/design-spec.md §1.4 (motion) + §1.5 (haptics).

## Do

1. Audit each §1.4 row against the code and fix drift in place (expected home in parens):
   - Card-stack bob: 4 s ease-in-out infinite, front ±6 px, backs ±4 px, phase-offset
     (`Sources/DesignSystem/CardStackHero.swift`).
   - Progress ring fill: from empty, ~1.2 s ease-out on appear
     (`Sources/DesignSystem/ProgressRing.swift`).
   - Progress bar grow: scaleX 0→1, ~1 s ease-out, anchored left
     (`Sources/DesignSystem/AnimatedProgressBar.swift`).
   - Check pop: scale 0 → 1.18 → 1, ~0.5 s spring (`Sources/DesignSystem/CheckPopView.swift`).
   - Wrong-answer shake: ~0.5 s (`Sources/DesignSystem/ShakeEffect.swift` + its Session usage).
   - Audio pulse ring: scale 0.85→1.7, opacity 0.6→0, 2.2 s, infinite
     (`Sources/DesignSystem/PulsingAudioButton.swift`).
   - Flame flicker: 2.2 s ease-in-out infinite (`Sources/DesignSystem/StreakPill.swift`,
     `Sources/Features/Progress/StreakCard.swift` — both must use the same treatment).
   - Card-to-card: off-LEFT / in-from-BELOW, spring ~0.35 s
     (`Sources/Features/Session/MCSessionView.swift`, `Sources/Features/Session/ReviewSessionView.swift`).
   - Session-complete check pop ~0.55 s after a 0.3 s delay
     (`Sources/Features/Session/SessionCompleteView.swift`).
2. Centralise durations: every animation duration/spring parameter in `Sources/` must come from
   the `Motion` enum (`Sources/DesignSystem/` tokens file). Add any missing constants to
   `Motion` (e.g. `completePop = 0.55`, `completePopDelay = 0.3`) rather than leaving inline
   numbers. Replace all numeric-literal durations in views accordingly.
3. Motion accessibility & energy conformance — for EVERY infinite/looping animation (card-stack
   bob, audio pulse ring, flame flicker):
   - gate it on `@Environment(\.accessibilityReduceMotion)` — when Reduce Motion is on, render
     the static end state (no loop); one-shot fills (ring/bar grow, check pop) may stay;
   - it must PAUSE when its screen/tab is not visible — the main shell keeps all four tabs
     alive, so drive each loop from a bool tied to visibility (`.onAppear`/`.onDisappear` or
     the shell's selected-tab state), never an unconditional `repeatForever` started at init.
4. Haptics conformance (§1.5): exactly these system feedbacks exist, nowhere else —
   - light impact on selecting an answer / tapping a grade button,
   - `.success` notification on a correct answer,
   - `.error` notification on a wrong answer.
   The CANONICAL mechanism is SwiftUI's `.sensoryFeedback(_:trigger:)` modifiers (the system
   manages generator lifecycle/prepare). Find every `UIImpactFeedbackGenerator` /
   `UINotificationFeedbackGenerator` / `.sensoryFeedback` usage; migrate stray UIKit generator
   call sites to `.sensoryFeedback` (or a single pinned helper if a non-view site genuinely
   needs one); remove extras; add any of the three that are missing at the pinned trigger
   points (Session screens). No custom haptic patterns (no Core Haptics).
5. Record the audit in the worklog entry for this task: a table of each §1.4 row →
   implementing file:line after the sweep, the full list of haptic call sites, and the
   Reduce-Motion/visibility gating point for each infinite animation.
