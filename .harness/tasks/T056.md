# T056 — NotificationsService: review-due local notifications, permission after first session

## Overview

Add the local-notification service that nudges ONLY when reviews are actually due ("12 cards are
slipping"), requests permission after the first completed session (never at launch), and
reschedules as the SRS forecast changes. Design reference:
.harness/docs/designs/design-spec.md §7.3; architecture:
.harness/docs/designs/architecture.md.

## Do

1. Create `Sources/Services/NotificationsService.swift`:
   - Protocol seam so tests never touch the real center:
     ```swift
     protocol UserNotificationCentering: AnyObject {
       func requestAuthorization() async throws -> Bool     // .alert + .sound + .badge
       func add(_ request: UNNotificationRequest) async throws
       func removePendingRequests(withIdentifiers ids: [String])
       func pendingRequests() async -> [UNNotificationRequest]
     }
     ```
     plus a thin `extension`-based conformance for `UNUserNotificationCenter`.
   - `final class NotificationsService` with
     `init(center: UserNotificationCentering, userDefaults: UserDefaults = .standard)` and:
     - `func requestPermissionIfNeeded() async -> Bool` — if the `Bool` flag under UserDefaults
       key **"notifications.asked"** is set, return the stored grant result without touching
       the center; otherwise call `requestAuthorization()`, persist both the asked flag and
       the result, and return it. It must hit the center AT MOST ONCE ever.
     - `func rescheduleReviewNotification(dueCount: Int, nextDueDate: Date?)` — always first
       `removePendingRequests(withIdentifiers: ["review-due"])`; then, ONLY if
       `nextDueDate` is non-nil and in the future and `dueCount > 0`, add one request with
       identifier **"review-due"**, title **"Enough"**, body **"<n> cards are slipping"**
       (`dueCount == 1` → "1 card is slipping"), and a `UNCalendarNotificationTrigger`
       (`repeats: false`) at `nextDueDate`. Never anything repeating, never a request when
       nothing will be due.
     - `func cancelAll()` — removes the pending "review-due" request.
     - `static func forecast(dueDates: [Date], now: Date) -> (count: Int, fireAt: Date)?` —
       pure helper: `fireAt` = the earliest due date strictly after `now` (nil result if none);
       `count` = how many `dueDates` are ≤ `fireAt` (already-due cards still count — they'll
       be due then too).
2. Register it in DI: in `Sources/App/AppServices.swift` add
   `let notifications: NotificationsService`, constructed with `UNUserNotificationCenter
   .current()` in `live()` and a no-op fake in `preview()` (define the tiny fake alongside
   `preview()` if none exists).
3. Wire the two recompute sites (pin: the forecast inputs are the `dueAt` values of
   `services.cardSRSStore` records whose `statusRaw != "new"` in owned decks —
   `services.entitlementStore.ownedDeckIds(catalog:)` — computed inline at each site, with
   `now = services.dateProvider.now`):
   - `Sources/Features/Session/SessionCompleteView.swift`: in `.onAppear` (this screen only
     appears after a completed session, satisfying "after the first completed session"), first
     `await services.notifications.requestPermissionIfNeeded()`, then compute the forecast and
     call `rescheduleReviewNotification(dueCount:nextDueDate:)`.
   - `Sources/App/RootView.swift`: observe `@Environment(\.scenePhase)`; when it becomes
     `.active` while in the `.main` phase, recompute + reschedule (no permission request here).
4. Add `Tests/NotificationsServiceTests.swift` with a `FakeNotificationCenter` recording calls.

## Done when

- `xcodebuild test` passes with at least: (a) two `requestPermissionIfNeeded()` calls hit the
  fake center exactly once and the second returns the persisted result (use a scratch
  `UserDefaults(suiteName:)`, cleaned in `tearDown`); (b)
  `rescheduleReviewNotification(dueCount: 12, nextDueDate: future)` removes "review-due" then
  adds exactly one non-repeating request with body "12 cards are slipping" and the right
  trigger date components; (c) `dueCount: 0` or nil/past date → remove happens, nothing added;
  (d) singular body "1 card is slipping"; (e) `forecast` returns the earliest FUTURE date with
  a count including already-due items, and nil when no future dates exist.
- No permission prompt at app launch (nothing in `EnoughApp`/onboarding calls it) — only
  `SessionCompleteView` requests, once ever.
- The pending notification reflects the latest forecast after each session completion and on
  app foreground.
