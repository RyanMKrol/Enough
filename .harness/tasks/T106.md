## Overview

Fix the multiple-choice learn session so it actually renders when started. Right now
`MCSessionView` shows a permanent blank white screen instead of the flashcard session, which makes
the entire learn flow (T047) unreachable from the UI — a confirmed regression reproduced on the
simulator.

## Do

In `Sources/Features/Session/MCSessionView.swift`, fix the broken lazy view-model initialization so
the session content renders. The view holds `@State private var viewModel: MCSessionViewModel?`
(line 6) and only builds it inside `.onAppear` on the `else`-branch `Color.white` placeholder
(lines 101–106); that state mutation never causes `content` to re-evaluate into the `if let
viewModel` branch (line 33), so the `fullScreenCover` is stuck on the blank placeholder forever.
Rework the initialization to a state-safe pattern so `viewModel` reliably becomes non-nil and the
populated branch renders — for example, build it in a `.task`/`.onAppear` attached to a **stable**
container view (not the branch that is about to be swapped out), or initialize it eagerly from the
`engine` + `services` the view already has (`MCSessionViewModel(engine:services:)`), being mindful
that `services` comes from `@Environment` and so is not available in `init`. Keep the change confined
to the session presentation layer; `MCSessionViewModel.init`/`loadCurrentCard` themselves are not
throwing or crashing (verified — this is a SwiftUI view-state/re-render bug, not a session-content
or empty-queue bug), so only touch `MCSessionViewModel.swift` if the chosen fix genuinely requires it.
Do not change how sessions are started from the call sites (`HomeView.startLearnSession`,
`DeckDetailView.startLearnSession`) — those are correct; the presented view is what's broken.

## Done when

Starting a multiple-choice learn session renders the real session UI — the question header, the
answer-option rows, and the `SessionChrome` (close button + progress bar) — instead of a blank white
screen, verified on the simulator from **both** entry points:

1. The **Home / Learn screen** Continue-card play button (`HomeView.startLearnSession`), and
2. The **Deck Detail** screen's "Continue learning" button (`DeckDetailView.startLearnSession`).

For each path, after tapping, the presented session view appears with its content (a card's target
prompt and tappable answer rows) and its close control — confirmed by the accessibility identifiers
`AXID.screenSessionMC` (`screen-session-mc`) and the session close control (`session-close`) both
being present/matchable in the running app, where before the fix they never matched. Tapping the
close control dismisses the session back to the presenting screen. This is a visual-render fix, so
it must be eyeballed on the simulator (default launch — no `-demo-state` needed; a "New" deck with
its seeded cards is sufficient), not merely a green build.
