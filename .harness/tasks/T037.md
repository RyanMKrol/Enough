# T037 — Debug menu shell: hidden entry (shake / launch argument) + empty sections

## Overview

Create the hidden "god mode" debug menu: a sheet listing registered debug sections, opened by
shaking the device or launching with `-debug-menu`. This task builds only the shell + registry
mechanism; later tasks (T038–T042, T057) each register their own row, so the registry's shape
here is a binding contract.

## Do

1. Create `Sources/Features/Debug/DebugRegistry.swift` with EXACTLY these types (later tasks
   depend on these names/signatures verbatim):

   ```swift
   struct DebugSection: Identifiable {
     let id: String
     let title: String
     let rows: [DebugRow]
   }

   struct DebugRow: Identifiable {
     let id: String
     let title: String
     var subtitle: String?
     let kind: DebugRowKind
   }

   enum DebugRowKind {
     /// Runs work and returns a human-readable result message, shown as a toast.
     case action((AppServices) async throws -> String)
     /// Integer value with -/+ controls; `label` formats the current value for display.
     case stepper(get: (AppServices) -> Int, set: (AppServices, Int) -> Void, label: (Int) -> String)
     /// Read-only value, recomputed each render.
     case info((AppServices) -> String)
   }

   enum DebugProviders {
     /// Every debug task appends exactly one `Debug<Name>.section` literal to this array.
     static let all: [DebugSection] = [
       DebugAbout.section
     ]
   }
   ```

2. Create `Sources/Features/Debug/DebugAbout.swift`: `enum DebugAbout` with
   `static let section = DebugSection(id: "about", title: "About", rows: [...])` containing two
   `.info` rows: "App version" (from `Bundle.main` `CFBundleShortVersionString` + build number)
   and "Active trip" (via `services` → `TripStore.activeTrip()`: "japan · weekend · day 2", or
   "none" — catch errors into "unavailable").
3. Create `Sources/Features/Debug/DebugMenuView.swift`: a `NavigationStack` + `List` rendering
   `ForEach(DebugProviders.all)` as sections; reads `@Environment(\.services)`. Row rendering by
   kind: `.action` → `Button` (run in a `Task`, catch errors into the message); `.stepper` →
   title + `Stepper` whose label shows `label(get(services))`; `.info` → `LabeledContent`.
   Action results show as a transient toast: `@State private var toast: String?` rendered as a
   capsule overlay at the bottom, auto-clearing after ~2s. Navigation title "Debug", a
   toolbar Done button dismissing the sheet, and a list footer "Debug menu — not user-facing.".
4. Create `Sources/App/ShakeDetector.swift`:
   - `extension UIDevice { static let deviceDidShakeNotification = Notification.Name("enough.deviceDidShake") }`
   - `extension UIWindow { open override func motionEnded(...) }` posting that notification on
     `.motionShake` (the standard SwiftUI shake-detection pattern), calling `super`.
   - `extension View { func onShake(perform action: @escaping () -> Void) -> some View }` via a
     `ViewModifier` using `.onReceive(NotificationCenter.default.publisher(for:))`.
5. In `Sources/App/MainShellView.swift` add `@State private var showDebugMenu = false`, apply
   `.onShake { showDebugMenu = true }` and `.sheet(isPresented: $showDebugMenu) { DebugMenuView() }`,
   and in `.onAppear` set `showDebugMenu = true` when
   `ProcessInfo.processInfo.arguments.contains("-debug-menu")`.
6. Do NOT wrap anything in `#if DEBUG` — the menu ships in all builds (hidden entry is enough).
7. Add `Tests/DebugRegistryTests.swift`: assert `DebugProviders.all` is non-empty, section ids are
   unique, the "about" section exists with 2 rows, and row ids within each section are unique.

## Done when

- Launching the app in the simulator with `-debug-menu` presents the Debug sheet over the main
  shell; it lists the "About" section with the app-version and active-trip rows, and the Done
  button dismisses it.
- Shaking the simulator (Device → Shake) while in the main shell presents the same sheet.
- `Sources/Features/Debug/DebugRegistry.swift` contains the exact types above so later tasks can append
  `Debug<Name>.section` entries to `DebugProviders.all` without further changes to the view.
- `Tests/DebugRegistryTests.swift` passes with the assertions in step 7.
