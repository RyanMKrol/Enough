## Overview

Add the App Store-required privacy manifest (`PrivacyInfo.xcprivacy`). Missing or wrong
manifests are an automated submission rejection (since May 2024). Enough collects nothing and
runs fully on-device, so the manifest is small — the work is declaring required-reason APIs
honestly.

## Do

1. Create `Sources/PrivacyInfo.xcprivacy` (Apple plist XML format) declaring:
   - `NSPrivacyTracking`: false.
   - `NSPrivacyTrackingDomains`: empty array.
   - `NSPrivacyCollectedDataTypes`: empty array (no analytics, no network, no accounts).
   - `NSPrivacyAccessedAPITypes`: one entry — category
     `NSPrivacyAccessedAPICategoryUserDefaults` with reason code `CA92.1` (the app reads/writes
     its own UserDefaults: debug day-offset, notification bookkeeping).
2. Ensure XcodeGen bundles it: confirm the `Sources` group picks it up as a resource for the
   Enough target (XcodeGen treats non-source files in a sources path as resources by default —
   verify in the generated project; if not, add an explicit `resources:` entry in `project.yml`).
3. Audit for other required-reason APIs and record the result in your worklog: grep Sources/ for
   `creationDate`, `modificationDate`, `contentModificationDateKey`, `systemUptime`,
   `ProcessInfo.processInfo.systemUptime`, `volumeAvailableCapacity`, `boottime`. Expected: no
   hits (system frameworks declare their own reasons). If any hit exists, add the matching
   category + reason code to the manifest.

## Done when

- `Sources/PrivacyInfo.xcprivacy` exists with exactly the declarations above, is valid plist XML
  (`plutil -lint` passes), and appears inside the built app bundle (verify via `./build_run.sh`
  then `ls` of the .app in build output, or a test asserting
  `Bundle.main.url(forResource: "PrivacyInfo", withExtension: "xcprivacy") != nil`).
- The required-reason grep audit is recorded in the worklog with zero unexplained hits.
