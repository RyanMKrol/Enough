## Overview
Enhance the existing debug 'Reset all data' row so that, after wiping state, it immediately re-routes the running app back to the first onboarding screen without a manual kill/relaunch.

## Do
The debug reset already wipes state via AppReset.wipeAll (Sources/Services/AppReset.swift) but its toast tells the tester to relaunch. Make the reset ALSO re-enter onboarding in place: after the wipe, drive AppState back to first-run state (mirror the in-place re-route in AppState.startNewTrip() in Sources/App/AppState.swift — cancel notifications, set phase = .onboarding — but with isReonboarding = false so it lands on the very first WelcomeView, not the country picker). The debug action closure in Sources/Features/Debug/DebugRegistry.swift currently only receives AppServices, so you must plumb AppState to the reset path: either widen the action to also take AppState and hand it in from DebugMenuView (Sources/Features/Debug/DebugMenuView.swift, presented from Sources/App/MainShellView.swift which has @Environment(AppState.self)), or have DebugMenuView perform the re-route after invoking the wipe. RootView (Sources/App/RootView.swift) already re-renders live off the @Observable AppState.phase, so no relaunch is needed. Update the toast so it no longer says 'relaunch the app'. Prefer enhancing the existing 'reset-all' row over adding a new one (pending owner confirmation).

## Done when
In the running app (simulator, via the debug menu), tapping the 'Reset all data' row wipes trip/entitlements/SRS/activity/day-offset AND the app immediately displays the first onboarding screen (WelcomeView) with no manual relaunch; the reset toast no longer instructs the user to relaunch.
