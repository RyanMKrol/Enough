# T042 — Debug setting: seed demo state (mid-trip fixture, also via launch argument)

## Overview

Create `DemoSeeder`, which wipes the store and writes a deterministic mid-trip fixture that makes
Home / Reviews / Progress closely match the design mocks (day 2 of a Tokyo weekend, 12 reviews
due, 6-day streak). It runs from a debug-menu row or the `-demo-state` launch argument — the
latter is what the project's visual-verification script uses to screenshot a populated app.

## Do

1. Create `Sources/Services/DemoSeeder.swift`:

   ```swift
   enum DemoSeeder {
     /// Wipes all persisted data (AppReset.wipeAll) then seeds the demo fixture below.
     /// `now` comes from the caller (services.dateProvider.now).
     static func seed(content: ContentStore, context: ModelContext, now: Date) throws
   }
   ```

   Seeded fixture (all card ids are taken IN FILE ORDER from
   `ContentStore.cards(forDeck:)` — "first N" means the first N cards of that deck's JSON):

   | What | Value |
   |---|---|
   | Trip | `TripProfileRecord`: countryId "japan", duration "weekend", scenarioIds ["eating-out", "getting-around", "greetings"], startDate = now − 1 day (so day counter = 2), accentRawValue "japan", isActive true |
   | Entitlements | one `EntitlementRecord`: productId "jp-weekend", kind "bundle" (resolves to decks jp-greetings, jp-ordering-food, jp-at-the-bar via the catalog) |
   | jp-greetings SRS | first 5 cards: review, intervalDays 30, repetitions 4, dueAt now + 20d (mastered) · next 18: review, intervalDays 7, repetitions 3, dueAt now + 4d · last 7: review, intervalDays 3, repetitions 2, dueAt now − 1h (DUE) |
   | jp-ordering-food SRS | first 5 cards: learning, intervalDays 1, repetitions 1, dueAt now − 1h (DUE) · next 3: learning, intervalDays 1, repetitions 1, dueAt now + 1d · remaining 10 cards: NO row (New) |
   | jp-at-the-bar / jp-getting-around | no rows (deck states New / not owned respectively) |
   | All SRS rows | easeFactor 2.5, lapses 0, lastReviewedAt now − 1d |
   | Activity | 6 `DailyActivityRecord` rows for the 6 consecutive days ending YESTERDAY (days now−1 … now−6), each: secondsStudied 620, cardsReviewed 12, cardsLearned 6 |

   Resulting derived numbers (what the UI should show): 12 due now (7 + 5), 5 mastered,
   8 learning, 38 words learned, streak 6 (today has no activity yet), 62 total minutes,
   ordering-food progress 8/18, greetings "Learned" (30/30).
2. Create `Sources/Features/Debug/DebugDemoSeed.swift`: `enum DebugDemoSeed` with
   `static let section = DebugSection(id: "demo", title: "Demo", rows: [...])` containing one
   `.action` row (id "seed-demo", title "Seed demo state", subtitle "Replaces ALL data with the
   design-mock fixture") calling `DemoSeeder.seed` with the services' content store, model
   context, and `services.dateProvider.now`, returning `"Demo state seeded — relaunch to see Home"`.
3. Append `DebugDemoSeed.section` to the `DebugProviders.all` array literal in
   `Sources/Features/Debug/DebugRegistry.swift` (one-line edit, keep existing entries; insert
   BEFORE the "Danger zone" section if present).
4. Create `Sources/App/LaunchArguments.swift`: `enum LaunchArguments { static func
   handle(services: AppServices) }` — when `ProcessInfo.processInfo.arguments` contains
   `-demo-state`, it runs `DemoSeeder.seed` synchronously. Call `LaunchArguments.handle` from
   `RootView`'s initializer (RootView lives in `Sources/App/`, inside this task's scope) so the
   seed completes BEFORE the first frame renders and a launch with that argument always shows
   the seeded Home screen.
5. Add `Tests/DemoSeederTests.swift` (in-memory ModelContainer, real bundled ContentStore, fixed
   `now`): after `seed`, assert (a) `DeckProgressService.totalDue()` == 12; (b)
   `StatsService.currentStreak()` == 6; (c) `TripStore.dayNumber(now:)` == 2; (d)
   `EntitlementStore.ownedDeckIds(catalog:)` == ["jp-greetings", "jp-ordering-food",
   "jp-at-the-bar"]; (e) seeding twice does not duplicate rows (wipe-first is effective:
   exactly 38 CardSRSRecords and 6 DailyActivityRecords exist).
