# T057 — Debug setting: notification test fire + pending-schedule inspector

## Overview

Add debug-menu tools for the review-due notification pipeline (T056's `NotificationsService`):
fire the review notification in 2 seconds on demand, and inspect what is currently scheduled.

## Do

1. Read `Sources/Services/NotificationsService.swift` (built by T056) first and use its existing
   seam: the service is constructed with a `UserNotificationCentering` protocol wrapping
   `UNUserNotificationCenter` (add/getPending/removeAll). If T056 did not ship such a seam, add
   the protocol + a conforming `UNUserNotificationCenter` adapter in
   `Sources/Services/NotificationsService+Debug.swift` WITHOUT changing the service's existing
   public API.
2. In `Sources/Services/NotificationsService+Debug.swift`, extend `NotificationsService`:

   ```swift
   extension NotificationsService {
     /// Schedules the normal review-due notification (same title/body copy path as the real
     /// one, using the CURRENT due count) with a 2-second UNTimeIntervalNotificationTrigger,
     /// identifier "debug.test-review-notification". Returns the body text it scheduled.
     func fireTestNotification(dueCount: Int) async throws -> String
     /// "2 pending · next ~in 4h" style one-liner from getPendingNotificationRequests;
     /// "none pending" when empty.
     func pendingSummary() async -> String
   }
   ```

3. Create `Sources/Features/Debug/DebugNotifications.swift`: `enum DebugNotifications` with
   `static let section = DebugSection(id: "notifications", title: "Notifications", rows: [...])`
   containing (a) an `.action` row (id "fire-test", title "Fire review notification in 2s",
   subtitle "Background the app to see the banner") that gets the current due count from
   `services.deckProgress.totalDue()` and calls `fireTestNotification(dueCount:)`, returning
   `"Scheduled: <body>"`; (b) an `.info` row (id "pending", title "Pending notifications") —
   `.info` closures are synchronous, so cache the latest `pendingSummary()` result in the enum
   via a refresh kicked off by the action row / first render, defaulting to "tap Fire to
   refresh". Keep it simple; exactness of the async bridge is not the point of the row.
4. Append `DebugNotifications.section` to the `DebugProviders.all` array literal in
   `Sources/Features/Debug/DebugRegistry.swift` (one-line edit; keep existing entries).
5. Add `Tests/NotificationDebugToolsTests.swift`: with a spy `UserNotificationCentering` (records
   added requests), assert `fireTestNotification(dueCount: 12)` adds exactly one request whose
   trigger is a `UNTimeIntervalNotificationTrigger` with `timeInterval` == 2, whose body contains
   "12", and whose identifier is "debug.test-review-notification"; and that `pendingSummary()`
   reports the spy's pending count.

## Done when

- With notification permission granted on the simulator, tapping "Fire review notification in
  2s" and backgrounding the app shows the review-due banner within a few seconds, with the same
  copy shape as the real notification ("12 cards are slipping" style).
- `Tests/NotificationDebugToolsTests.swift` passes with the assertions in step 5.
