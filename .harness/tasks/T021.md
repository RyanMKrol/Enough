# T021 ‚Äî SRS queue builder: review ordering, learn chunking, miss re-queueing

## Overview

Add the pure helpers that decide WHICH cards enter a session and in what order: the due-review
queue (soonest-due first), the fixed-size learn batch (design-spec.md ¬ß6: sessions are 12 cards),
and the re-queue offset used when a learner misses a card mid-session.

## Do

1. Create `Sources/SRSKit/SRSQueueBuilder.swift` (`import Foundation` only) with:

   ```swift
   struct QueueEntry: Equatable {
     let cardId: String
     let deckId: String
     var state: SRSState
   }
   enum SRSQueueBuilder {
     static func reviewQueue(from entries: [QueueEntry], now: Date, limit: Int? = nil) -> [QueueEntry]
     static func learnBatch(newCardIds: [String], size: Int = 12) -> [String]
     static func requeuePosition(queueLength: Int) -> Int
   }
   ```

2. `reviewQueue`: keep only entries where `SRSEngine.isDue(entry.state, now: now)` (T020), sort
   by `dueAt` ascending (most-overdue first; two equal `dueAt`s keep input order ‚Äî use a stable
   sort or `sorted(by:)` on the index-paired array), then apply `limit` via `prefix` when non-nil.

3. `learnBatch`: `Array(newCardIds.prefix(size))` ‚Äî preserves catalog/deck-file order; no
   shuffling (the deck author's order is the pedagogical order).

4. `requeuePosition(queueLength:)` = `min(queueLength, 3)`. Meaning, pinned here and consumed by
   `SessionEngine` in T023: when a card is answered wrong, it re-enters the remaining queue this
   many positions after the current card (`queueLength` = number of cards remaining AFTER the
   current one), so a miss comes back quickly but not immediately.

5. Create `Tests/SRSQueueBuilderTests.swift` (XCTest) with fixed dates
   (`Date(timeIntervalSince1970:)` arithmetic only).

## Done when

  assert `reviewQueue` returns exactly the three due entries ordered
