## Overview

Finish price localization once real StoreKit exists: every user-facing price AND savings string
comes from the App Store's localized product data, so non-UK users see correct storefront
prices. Kills the Guideline 3.1.x hardcoded-price rejection risk left by the stub era.

## Do

1. Extend `PurchaseProviding` (in `Sources/Services/`) with
   `func price(productId: String) -> Decimal?` alongside the existing `displayPrice(productId:)`.
   StoreKit implementation: `product.price`; stub: catalog `priceGBP` as Decimal.
2. Add localized savings/summary helpers to the StoreKit-backed path: bundle savings computed
   from `Product.price` Decimals and formatted with `product.priceFormatStyle` (never
   `NumberFormatter(en_GB)`). Expose via a small
   `LocalizedPricing` helper in `Sources/Services/` that takes the purchase service + catalog
   and produces: unit price string, bundle savings string ("save …"), and plan total string —
   falling back to `PricingCalculator` only when products are unavailable (stub/preview).
3. Swap the consumers: `BrowseView`/`BrowseViewModel` savings lines and the Plan screen
   (`PlanViewModel`) totals/savings/summary strings route through `LocalizedPricing`.
4. Tests: with the stub (deterministic), `LocalizedPricing` falls back to `PricingCalculator`
   output verbatim (pinned fixture numbers still hold); with an injected fake product-price
   table (e.g. USD), savings math uses the injected Decimals (assert "$" formatting via a fixed
   `Locale(identifier: "en_US")` format style in the fake).

## Done when

- Grep gate: no `NumberFormatter` currency usage and no "£" literals in Sources/Features/;
  every price/savings string in Browse and Plan flows through `LocalizedPricing`.
- Stub-mode UI output is byte-identical to before (fixture tests unchanged and green).
- With StoreKit products loaded (StoreKitTest session), Browse shows `product.displayPrice`
  values.
