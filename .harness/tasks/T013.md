## Overview

Add the app-wide clock abstraction: a `DateProvider` protocol with a real system implementation
and an offsettable debug implementation (used later by the debug menu's "time travel" to test
spaced-repetition scheduling). Everything that needs "now" takes a `DateProvider` — nothing else
in the app should call `Date()` directly after this lands.

## Do

1. Create `Sources/CoreKit/DateProvider.swift`:

   ```swift
   protocol DateProvider {
     var now: Date { get }
   }

   struct SystemDateProvider: DateProvider {
     var now: Date { Date() }
   }
   ```

2. Create `Sources/CoreKit/AdjustableDateProvider.swift`:

   ```swift
   /// System time shifted by a persisted whole-day offset (debug "time travel").
   final class AdjustableDateProvider: DateProvider {
     init(userDefaults: UserDefaults = .standard)
     var dayOffset: Int { get set }   // persisted immediately on set
     var now: Date { get }            // Date() + dayOffset days
   }
   ```

   - The offset is stored in the provided `UserDefaults` under the exact key
     `"debug.dayOffset"` (define it as `static let offsetKey = "debug.dayOffset"`).
   - `init` reads any previously persisted offset so the value survives relaunch.
   - `now` = `Calendar.current.date(byAdding: .day, value: dayOffset, to: Date())` (fall back to
     `Date()` if the calendar call returns nil).

3. Create `Tests/DateProviderTests.swift` (XCTest, class `DateProviderTests`). Tests must be
   hermetic: build each `AdjustableDateProvider` with a throwaway
   `UserDefaults(suiteName: "DateProviderTests")!`, and call
   `UserDefaults().removePersistentDomain(forName: "DateProviderTests")` in `tearDown` so no
   state leaks between runs. NEVER use `UserDefaults.standard` in tests.

Conventions: single app target (tests use `@testable import Enough`), Swift 5 / iOS 18, 2-space
indent, SwiftLint strict (lines ≤ 130).

## Done when

- `Tests/DateProviderTests.swift` asserts:
  - `SystemDateProvider().now` is within 5 seconds of `Date()`.
  - A fresh `AdjustableDateProvider` (empty suite) has `dayOffset == 0` and `now` within
    5 seconds of `Date()`.
  - After `dayOffset = 3`, `now` is ~3 days ahead: the interval `now.timeIntervalSince(Date())`
    is within 5 seconds of `3 * 86_400` (use a tolerance assertion, not exact equality).
    Negative offsets work symmetrically (`dayOffset = -2` → ~2 days behind).
  - Persistence: set `dayOffset = 5` on one instance, create a SECOND instance with the SAME
    suite, and assert its `dayOffset == 5` — proving the value was written under
    `"debug.dayOffset"` (also assert `suite.integer(forKey: "debug.dayOffset") == 5`).
- No file outside `Sources/CoreKit/` and `Tests/` changes.
